# Findings - 2026-05-03

## P1 - Clear live Chroma documents when live cache expires or patches change

Files:

- `src/valocoach/retrieval/cache.py`
- `src/valocoach/retrieval/retriever.py`
- `src/valocoach/retrieval/patch_tracker.py`

`get_cached()`, `invalidate_volatile()`, and `purge_expired()` delete rows from
the SQLite `meta_cache` table, but no code deletes the corresponding documents
from the `valocoach_live` Chroma collection. `retrieve_static()` searches
`valocoach_live` on every coach query, so a scraped live meta page can continue
grounding the LLM after its 12-hour TTL expires or after a patch invalidates
volatile cache entries.

Impact:

- Patch-day live meta can keep appearing in grounded context after it should be
  invalidated.
- SQLite and ChromaDB stop representing the same freshness state.
- This contradicts the AGENTS.md rule that live content is TTL-managed via
  `meta_cache`.

Recommended fix:

- Store enough Chroma metadata to target live entries by `source`, `ttl_tier`,
  and optionally `expires_at`.
- On `invalidate_volatile()` and `purge_expired()`, delete matching documents
  from `LIVE_COLLECTION`, or gate `LIVE_COLLECTION` search results against
  currently valid SQLite cache rows.
- Add tests proving expired or volatile live documents no longer appear in
  `retrieve_static()`.

## P1 - Filter round-level stats with the same match set as aggregate stats

File: `src/valocoach/cli/commands/stats.py`

The stats command filters `rows` by period, agent, map, and result before
rendering aggregate stats, but the round-level section calls
`analyze_rounds(data.full_matches, player.puuid)` on the full loaded set. A
command like `valocoach stats --agent Jett --map Ascent --period 7d` can show
overall ACS/KD for filtered Jett Ascent matches while KAST, clutch, trade, and
side win rates come from all loaded competitive matches.

Impact:

- The stats view can silently mix scopes within one screen.
- User-facing round-level metrics are misleading under common filters.
- This violates the project rule that stats must not silently corrupt
  user-visible analysis.

Recommended fix:

- Build `filtered_match_ids = {mp.match_id for mp in filtered}`.
- Analyze only `m for m in data.full_matches if m.match_id in filtered_match_ids`.
- Add integration tests for `--agent`, `--map`, `--period`, and `--result`
  proving the round-level denominator follows the filtered rows.

## P2 - Make LLM preflight provider-aware

Files:

- `src/valocoach/cli/app.py`
- `src/valocoach/cli/commands/interactive.py`
- `src/valocoach/llm/provider.py`

`stream_completion()` supports `ollama/`, `anthropic/`, and `openai/` model
prefixes, but both `valocoach coach` and `valocoach interactive` always call
`check_ollama()` before invoking the provider. Setting `ollama_model` to a
cloud LiteLLM model therefore still fails unless local Ollama is reachable and
has a local model pulled.

Impact:

- The "provider swap is one config line" architectural decision is not true at
  the CLI boundary.
- Future Claude/OpenAI usage is blocked by an unrelated Ollama preflight.

Recommended fix:

- Rename the setting to a provider-neutral `llm_model` or add a compatibility
  alias.
- Run Ollama preflight only for models resolved to the Ollama provider.
- Add tests for a non-Ollama model prefix that assert no Ollama probe occurs.

## P2 - Apply round-count thresholds in aggregate reliability flags

Files:

- `src/valocoach/stats/calculator.py`
- `src/valocoach/stats/constants.py`
- `src/valocoach/cli/formatter.py`
- `src/valocoach/coach/context.py`

`SAMPLE_THRESHOLDS` defines both match and round floors, and `_check_threshold()`
honors both. The main rendering path uses `reliability_flags()`, which checks
only `stats.matches`. For example, 15 short matches with 150 total rounds clear
ACS/ADR in the formatter even though the constants require 200 rounds.

Impact:

- Warnings disappear for samples that the project explicitly classifies as
  unreliable.
- The coach context can treat thin data as reliable because it uses the same
  flags.

Recommended fix:

- Update `reliability_flags()` to evaluate both `stats.matches` and
  `stats.rounds` through `SAMPLE_THRESHOLDS`.
- Add tests with enough matches but too few rounds for ACS, ADR, KD, HS%, FB/FD,
  and econ rating.

## P2 - Stop printing secrets in `config show`

File: `src/valocoach/cli/app.py`

`valocoach config show` prints `Settings.model_dump()` directly, which includes
`henrikdev_api_key` when configured. This can leak the API key into terminal
scrollback, shared screenshots, support logs, and CI output if the command is
used for debugging.

Impact:

- Local API keys are exposed by a normal CLI command.
- This conflicts with the project's "Never commit API keys" posture by making
  accidental disclosure easier.

Recommended fix:

- Redact sensitive fields before rendering, for example
  `henrikdev_api_key = "***" if present`.
- Add a test that config output never contains the configured key string.

## P2 - Resolve environment variable naming drift

Files:

- `src/valocoach/core/config.py`
- `README.md`
- `.env.example`
- `tests/test_config.py`

`Settings` uses `env_prefix=""`, and tests assert unprefixed variables such as
`OLLAMA_MODEL`. README documents prefixed variables such as
`VALOCOACH_OLLAMA_MODEL`. `.env.example` uses the unprefixed form. The project
currently has two incompatible contracts.

Impact:

- Users who follow README shell exports get no override.
- Tests validate a different behavior than the public documentation.

Recommended fix:

- Choose one contract. Prefer `env_prefix="VALOCOACH_"` for shell environment
  safety, while optionally supporting legacy unprefixed names during a
  transition.
- Update README, `.env.example`, and tests together.

## P2 - Move Rich progress output out of the data layer

File: `src/valocoach/data/sync.py`

The `data` package is supposed to import only `core` internally, but
`data.sync` imports Rich `Console` and `Progress` and writes user-facing output
directly. This couples sync orchestration to the CLI presentation layer and
makes the future web tracker reuse the CLI progress dependency by default.

Impact:

- The module boundary rules are already broken in a core data path.
- Reusing sync from a web worker or service requires carrying Rich UI behavior
  or patching it out.

Recommended fix:

- Keep sync orchestration in `data`, but expose structured progress events or a
  callback protocol.
- Render Rich progress bars only in `cli`.
- Add a small import-boundary test so `data`, `stats`, `retrieval`, and `core`
  cannot accidentally depend on UI packages.
